# Roles and Personas Checklist

## Purpose
This checklist helps teams verify that roles are clearly assigned and responsibilities are understood at project initiation and planning stages. Use this document to ensure team clarity and accountability before starting execution.

## When to Use
- During project initiation (after creating the Project One-pager)
- At project kickoff meetings
- When onboarding new team members
- During team restructuring or role changes

## Role Assignment Verification

### Core Delivery Roles
- [ ] **Developers** are identified and assigned
  - [ ] Technical skills match project requirements
  - [ ] Capacity and availability confirmed
  - [ ] Clear understanding of responsibilities (coding, testing, code reviews)
  
- [ ] **Product Manager** is assigned
  - [ ] Has authority to prioritize backlog
  - [ ] Understands success metrics and business goals
  - [ ] Clear ownership of product vision and roadmap
  
- [ ] **Project Manager** is assigned
  - [ ] Responsible for timeline and delivery coordination
  - [ ] Has bandwidth to manage project scope and dependencies
  - [ ] Clear ownership of project documentation and status reporting

### Agile and Process Roles
- [ ] **Scrum Master** is assigned (if using Scrum/Agile)
  - [ ] Responsible for facilitating ceremonies
  - [ ] Empowered to remove impediments
  - [ ] Committed to coaching team on agile practices

### Design and User Experience
- [ ] **UX Designer** is assigned (if project involves UI/UX)
  - [ ] Has access to users for research and testing
  - [ ] Clear deliverables (wireframes, prototypes, design specs)
  - [ ] Aligned with design system standards

### Infrastructure and Operations
- [ ] **DevOps Engineer** is assigned
  - [ ] Responsible for CI/CD pipeline and infrastructure
  - [ ] Clear SLAs for deployment and environment support
  - [ ] Monitoring and alerting systems in place

### Customer-Facing Roles
- [ ] **Support/Customer Success** is identified
  - [ ] Has access to product for testing and troubleshooting
  - [ ] Process for escalating customer issues to engineering
  - [ ] Responsible for documentation and help articles

### Security and Compliance
- [ ] **Security Lead** is consulted (if security requirements exist)
  - [ ] Security requirements documented
  - [ ] Security review scheduled in project timeline
  - [ ] Clear process for vulnerability remediation

### Business and Leadership
- [ ] **Business Stakeholder(s)** are identified
  - [ ] Has authority to approve budget and scope
  - [ ] Agrees to communication cadence (e.g., weekly/monthly reviews)
  - [ ] Clear escalation path for decisions

## Responsibility Clarity

### Communication Plans
- [ ] Communication frequency and channels defined for each role
- [ ] Stakeholder update cadence agreed upon
- [ ] Meeting schedule established (standups, planning, reviews, retros)
- [ ] Escalation paths documented for blockers and decisions

### Cross-Role Collaboration
- [ ] Dependencies between roles are mapped
- [ ] Handoff points are clear (e.g., design → development, development → QA)
- [ ] Shared tools and processes agreed upon (project boards, repositories, docs)
- [ ] Conflict resolution process established

### Documentation and Onboarding
- [ ] All team members have access to project documentation
- [ ] Roles and responsibilities are documented in the Project One-pager
- [ ] New team members have an onboarding plan
- [ ] Contact list with role assignments is maintained

## Decision-Making Authority

- [ ] Clear who makes final decisions on:
  - [ ] Product priorities and feature scope
  - [ ] Technical architecture and design decisions
  - [ ] Resource allocation and timeline changes
  - [ ] Security and compliance requirements
  - [ ] Budget and staffing decisions

## Sign-Off

Once all applicable items are checked and roles are clear:

- [ ] **Project Manager** confirms all roles are assigned and documented
- [ ] **Product Manager** confirms alignment on product priorities and success metrics
- [ ] **Team Lead/Engineering Manager** confirms team capacity and availability
- [ ] **Business Stakeholder** approves team structure and budget
